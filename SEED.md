# Conceptual Specification of our New App 

We will completely strip down the chat sidebar entirely, everything will be centered in the 3D projector view. We will use Milkdown to replace the 2D knowledge panels, where we will stream our instant quantized model outputs to while simultaneously rendering the markdown knowledge panel dynamically. We are essentially building a billboarded version of the left-hand side (rendered markdown) from the milkdown playground: https://milkdown.dev/playground. The billboard should be appropriately sized (not too big, not too small, takes up maybe a sixth or seventh of the full 2D UI area), and scrollable within its billboard. We will build in a simple prompt panel at the very bottom of the billboard which persists at the very bottom and is independent of the scrollable markdown response. This chat panel should also be scrollable. The chat panel must have file and chat linking features, similar to the way that Claude and Google Antigravity prompt panels have special hooks for local 'things' like files (which we will also extend to referencing other chats in the same 3D projector). For referencing other chats and entities in the projector, we will add a feature that displays auto-suggest with the '@' symbol as a mini, scrollable dropdown menu that the user can select which reference to make to the other entities. For backslashes, instead of calling tools, we will apply a retrieval query search to other entities in the projector and return the mini-scrollable ranked suggestions of entities to select from the projector. The retrieval rule is such that everything after the backslash is included as a retrieval query (to avoid spacing issues). Then, the user can see the pop-up mini-scrollable with ranked results. When a user clicks a result, the mini-scrollable closes and the retrieval query text is highlighted with a preceding special sprite indicator on the set of rendered tokens (i.e. a crystal ball emoji with specially colored violet text). When a user enters a prompt into the milkdown billboard chat, the prompt renders the context of referenced nodes in an efficient recursive chunking manner with a hard limit on context windows of around 256 tokens. For that matter, all nodes in our projector are either chats or responses. The context links of the prompt are inserted into the hermes chat context inline with the rest of the chat context, so as to ensure that streaming chats requires minimal new information back and forth, and context links are engineered as predicates of the rendered prompt to the model, but are hidden to the markdown because their explicit links to other nodes are already evident in the 3D gui. Since our context prior in our rendered prompts (prior to the user's actual message) requires strict token limits of 256 tokens max for all items in the context links, we do some more advanced approaches. For the full set of nodes that are linked, we assume that the markdown structure of the LLM's responses are such that we can render their real structures as recursive trees delimited by things like headings, subheadings, bullet points, numbers, and singular table rows with their headers. So, for each linked node in the context, regardless of size, we apply this recursive chunking scheme over the full markdown itself. So, each chat and response gets a general embedding, and then recursive chunking embeddings when rendering prompt contexts to the LLM. We update mutated fields dynamically in fact, and essentially apply embeddings over only recursive fields, then aggregate embeddings for full chat and response nodes (without their external context links, since they already have their own recursive chunk embeddings from markdown tree structures). We apply a hard token limit window for each prompt's linked context nodes, and then perform a full unified retrieve and rank over the full set of linked recursive context fields as described. Then, we descend-iterate over our retrieval ranks and add the total token cost until the hard-context window limit is satisfied. We treat code artifacts as their own chunks as well. Anything with triple-` notation above and below, that one thing that SLMs do. We dynamically update embeddings over their recursive fields if they are modified, and then update the full markdown node embeddings of prompts and responses for global retrieval. So, each milkdown billboard its node embedding is the global embedding of all recursive chunks in the same scrollable chat. I should also note that since retrieval happens in the new way we just specified, we can also remove the retrieval sidebar from previous apps and just stick to the 3D projector as the main source for our GUI, with 2D billboards as described coming out of that. Note that since we use a kuzu vector-graph database, we should store the details of the original prompts and responses such that they directly link to their various embeddings (global and recursive). We could track the progression and structure of the markdown chats with their own links to other nodes such that we have full links over chat history and context links while storing each recursive chunk in their native chunk form with graph links, which are then projected back to their original full specialized markdown and context rendering in the GUI. Note that when we do retrieval with the '/' usage, we are actually going to end up showing the recursive chunk results (fully) in the mini-popup-scrollable. 

The recursive tree embeddings are what are shown as nodes in the 3D projector. However, when the user overs over any of the chunk nodes, the same billboard for the full chat opens with the scroll initialized to make centrally visible the chunk that the user is hovering over. This enables a global perspective over chats via fine-grained 'feeling out' over the semantic space of recursive markdown chunks. The global vector embeddings are also nodes that are twice as large as their chunk nodes. Highlighting a document embedding node should also highlight the other recursive chunk nodes, but the scrolling here is non-selective. As might already be obvious, when a user hovers over a node and sees the chat document as scrollable milkdown billboard, and then clicks into the billboard, this billboard is persisted as 'open' yet still dependent on the location of the 3D nodes. Thus, the billboard follows a kind of projective distance metric such that when the user interacts with the billboard, it is flown to and made to be fully in-view. But, right-clicking on the panel unsticks the invisible fly-to-view-and-focus link, allowing the user to continue navigating the 3D GUI while the opened node billboard remains visible in the distance as the milkdown panel. When clicked and stuck, these billboards are also completely draggable. Left-clicking does not close the billboard anymore, but does open it and persist it. Right-clicking closes it, returning the system to the show-when-hovered mode. Additionally, ensure that the billboards are *exactly* as the left hand rendered view in the milkdown playground.

This answers context control a bit more directly since we are selecting chunks and not full documents, so this can precede other design choice on rendering chunks to the context with context window limits, sticking to a simple rank-and-render as previously described, where the chunks of like documents are rendered/grouped as such in the context. Also, when this rendering on per-document is happening, we also track the preceding headings and subheadings that contain the chunk as well.
