# Objective Facts & Verbatim Requirements

The following statements are captured verbatim as objective facts regarding the requirements for the Solvolos 3D Milkdown App interactions:

1. We begin with an empty app. We double-click on the empty background to get a node that is represented as a milkdown billboard panel opened (meaning only the billboarded panel in 3D without a sphere). Because we double-clicked, the billboard panel is 'stuck' in its place to the 2D editor, and follows with the user as they navigate and click outside the billboard. 
2. We click and drag the milkdown billboard around the 2D ui. We type in some markdown fields in the milkdown editor that are instantly rendered in their true-markdown display. This means we obtain things like nested lists, enumeration, tables within heading subtrees, etc instantly as we interact with the milkdown editor itself. This must be verified somehow in playwright. 
3. After typing some recursive fields into our milkdown and seeing an instant rendering of the tree-structured document, we then interact with the LLM in a chat panel below. This panel is independent of the scrollable milkdown editor. This panel is tall enough for three lines of text to be written. 
4. We ask a simple natural-language question to our milkdown chat, and we get in return some response that is appended *after* our original updates to the document. This is tested for by ensuring the field structure of the original markdown's subgraph is isomorphic in its post-second-response-recursive-semantic-update DOM diff. Please capture this as requirements and verbatim statements of objective fact of requirements details everywhere. 
5. If we get a response in the form of markdown from the SLM, the recursive field chunks of the response's markdown and/or paragraphs and code artifacts are diff-updated inline with the previous milkdown that was manually typed in and instantly rendered. This means that the tree structures of the original markdowns from the user and SLM are isomorphic as subtrees to the updated semantically-composed set of recursive tree fields from the history of inputs from both the user and responses from the SLM.
6. As we interact with the milkdown panel, a debounce update to the panel's embedding and projector state triggers to send the finally rendered milkdown payload (as seen in the milkdown editor, but in original markdown format) to the quantized embedder, which rewrites the embedding state in the kuzu DB, which then triggers a delta update in the UMAP projector which also persists coordinate updates to the kuzu, and then finally sends the updated coordinate back to the 3D UI. 
7. As we interact with the app, the same milkdown billboard, when right-clicked, is flown back to its original 3D note position, while still being billboarded as a chat panel (but from a distance).
8. No duplicates or diffeomorphisms of tree structures that should be isomorphic in milkdown editors at any time, this should be tested for continuously and repeatedly throughout interactions with our app.
9. The UMAP is 6-dimensional, three spatial and three HSV (color) dimensions, which are rotated continuously within the same UMAP projector model in the exact same way as is done in ./legacy_wips/tracker. This is the non-copy version of the frontend's 3D projector layout. This should be completely integrated into our frontend design, at least the 3D projector part and not the 2D search retrieval or other 2D sidebar panels. And instead of spheres, we rotate colors on the highlighting of the text within the black-slate panels that rotate exactly as per the old tracker's frontend code with UMAP 6D embeddings and rotations at the exact same speed. 
10. The 2D chat panel at the bottom of billboards adaptively grows downward to fully capture the user's input, but collapses to its regular three-line containment when the user isn't either hovering or typing in the panel.
11. When linking through retrieval to other recursive node chunks with the forward-slash operator ("/"), the mini-pop-up panel is shown as a scrollable set of chunk results to the *left* of the milkdown billboard.
12. When a retrieval result is scrolled over and selected, its original chunk contents (with preserved recursive markdown fields and structures) are shown directly as the highlighted violet text with a crystal ball in front of the text, acting as a markdown hyperlink to the in-context node path in the Kuzu DB. This retrieved chunk text itself is what is input to the chat panel window to provide clear verbose context to the agent.
13. When this crystal-ball text is hovered over, the corresponding milkdown panel from its original source is temporarily shown in 2D beside the milkdown panel on the *right* side, scrolled to view the referenced chunk.
14. If the user clicks the highlighted text, the referenced milkdown billboard revealed to the right of the original is also 'stuck' (pinned). There is no interference between retrieving on the left and bringing in explicit links on the right.
15. When a node is clicked and sticked to the 2D, its text color immediately changes to pure white, overriding any UMAP color rotations.
16. There must be an essentially strict architectural copy-paste between the legacy tracker (`legacy_wips\tracker - Copy\tracker`) and our 3D projector's node position and color layout/rotations. This includes UMAP XYZ and HSV coordinates being constantly globally normalized (tared) to reside at the origin of the projector's 3D viewer.
17. All of these factors must be included in full-blown (and real-model) Playwright unit tests on an actual open browser with the DB running in the background, without any mocks, using quantized models and UMAP re-projection feedbacks from embeddings computed on markdown debounce mutations.
